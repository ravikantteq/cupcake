# Product Requirements Document (PRD)
# Cupcake | Backyard - Kafka Producer Testing Tool

## Project Overview

**Project Name**: Cupcake | Backyard  
**Date**: July 25, 2025  
**Version**: 1.0  
**Status**: Implemented  

### Problem Statement
The user needed a comprehensive Kafka producer testing tool with a clean, maintainable architecture that could:
- Provide a web UI for easy Kafka message publishing
- Offer REST API endpoints for programmatic access
- Include comprehensive testing capabilities
- Support future extensibility without overengineering

### Solution Overview
A dual-component system consisting of:
1. **Frontend**: Angular-based web UI (`cupcake_ui/`)
2. **Backend**: Go-based REST API server (`backyard/`)

## Architecture & Technology Stack

### Backend (Go)
- **Framework**: Gin web framework
- **Kafka Client**: Confluent Kafka Go client
- **Documentation**: Swagger/OpenAPI with auto-generation
- **Architecture Pattern**: Clean Architecture with layered separation

### Frontend (Angular)
- **Framework**: Angular 19
- **HTTP Client**: Angular HttpClient
- **Styling**: CSS with responsive design
- **Components**: Standalone components architecture

### Development Tools
- **Build Automation**: Makefile
- **Containerization**: Docker + Docker Compose
- **Documentation**: Comprehensive README + inline docs
- **Testing**: Unit and integration tests

## Feature Requirements

### Core Features (Implemented ✅)

1. **Kafka Producer API**
   - REST endpoint: `POST /api/kafka/publish`

2. **Web User Interface**
   - Form-based message publishing

6. **Testing Framework**
   - Unit tests for core logic

## Implementation Details

### API Endpoints

#### 1. Publish Message
- **Endpoint**: `POST /api/kafka/publish`
- **Purpose**: Publish JSON messages to Kafka topics
- **Request Body**:
  ```json
  {
    "broker": "localhost:9092",
    "topic": "test-topic", 
    "key": "optional-key",
    "value": "{\"data\": \"your message\"}"
  }
  ```
- **Response**: Success/error with detailed messaging

#### 2. Health Check
- **Endpoint**: `GET /health`
- **Purpose**: Monitor backend service status
- **Response**: Service health status and metadata

### Frontend Components

#### 1. Kafka Producer Component
- Form-based message publishing interface
- Real-time validation and error handling
- Success/failure feedback display
- Backend connectivity testing

#### 2. Main App Component
- Application shell and navigation
- Service status monitoring
- Links to documentation and APIs

### Testing Strategy

#### Backend Tests
- **Unit Tests**: Core Kafka producer logic
- **Integration Tests**: API endpoint functionality (Kafka-dependent)
- **Validation Tests**: Input validation and error scenarios
- **Coverage**: Core business logic and API contracts

#### Frontend Tests
- **Component Tests**: UI component behavior
- **Service Tests**: HTTP service communication
- **Integration Tests**: End-to-end user workflows

### Development Workflow

#### Local Development
1. **Backend**: `make backend` (runs on :8080)
2. **Frontend**: `make frontend` (runs on :4200)
3. **Combined**: `make dev` (starts both services)

#### Testing
1. **Backend Only**: `make test-backend`
2. **Frontend Only**: `make test-frontend`
3. **Full Suite**: `make test`

#### Production Build
1. **Backend**: `make build-backend`
2. **Frontend**: `make build-frontend`
3. **Combined**: `make build`

### Docker Support

#### Individual Services
- Backend: Dockerfile with multi-stage build
- Frontend: Nginx-based production container

#### Full Stack
- docker-compose.yml with Kafka, Zookeeper, Backend, Frontend
- Kafka UI for monitoring (optional)
- Network configuration for service communication

## User Experience

### Primary User Flow
1. User opens web interface at `http://localhost:4200`
2. Fills in Kafka broker, topic, key (optional), and message value
3. Clicks "Publish Message" to send to Kafka
4. Receives immediate feedback on success/failure
5. Can check backend health and view API documentation

### Alternative Flows
- Direct API usage via curl/Postman
- Swagger UI for interactive API testing
- Health monitoring for service status

## Technical Decisions & Rationale

### Technology Choices

1. **Go for Backend**
   - **Rationale**: High performance, excellent Kafka client library, strong typing
   - **Benefits**: Fast compilation, good concurrency, simple deployment

2. **Angular for Frontend**
   - **Rationale**: User requested Angular, mature framework
   - **Benefits**: TypeScript support, robust ecosystem, good testing tools

3. **Gin Web Framework**
   - **Rationale**: Lightweight, fast, good middleware support
   - **Benefits**: Easy routing, built-in JSON handling, minimal overhead

4. **Swagger Documentation**
   - **Rationale**: Auto-generated docs, interactive testing
   - **Benefits**: Always up-to-date docs, reduces documentation overhead

### Architecture Decisions

1. **Clean Architecture Pattern**
   - **Rationale**: Maintainable, testable, extensible
   - **Implementation**: Separate layers for API, business logic, and infrastructure

2. **Standalone Components (Angular)**
   - **Rationale**: Modern Angular pattern, better tree-shaking
   - **Benefits**: Improved performance, cleaner dependencies

3. **Error-First Design**
   - **Rationale**: Robust error handling throughout the stack
   - **Implementation**: Comprehensive error types and user-friendly messages
